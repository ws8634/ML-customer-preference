# 麦当劳客户预测系统 - 架构文档

## 核心文件说明

| 文件名称 | 核心职责 | 主要函数 | 依赖关系 |
|---------|---------|---------|---------|
| **model_training.py** | 模型训练与持久化 | - `_preprocess()`: 数据预处理<br>- `_create_ensemble_model()`: 创建集成模型<br>- `train_model()`: 训练并保存模型<br>- `load_model()`: 加载已训练模型 | 依赖pandas、scikit-learn、joblib |
| **model_evaluation.py** | 模型性能评估 | - `_preprocess()`: 数据预处理<br>- `load_and_split_data()`: 数据加载与划分<br>- `evaluate_model()`: 计算评估指标<br>- `print_evaluation_results()`: 打印评估结果 | 依赖model_training.py、pandas、scikit-learn |
| **api_service.py** | REST API服务 | - `/predict`: 预测端点<br>- `/health`: 健康检查<br>- `/model-info`: 模型信息 | 依赖model_training.py、flask |

## 架构决策记录

### 决策1：模型选择 - 随机森林 + 梯度提升树
**问题**：原单一决策树模型存在什么问题？
- 单一决策树容易过拟合，泛化能力有限
- 决策边界过于陡峭，对数据波动敏感
- 无法有效捕捉数据中的复杂模式

**可选方案评估**：
- 方案A：仅使用随机森林 - 抗过拟合好，但精度可能不足
- 方案B：仅使用梯度提升树 - 精度高，但可能过拟合
- 方案C：集成两种算法 - 优势互补，鲁棒性和精度兼顾

**最终决策**：选择方案C，软投票集成
- **随机森林**：抗过拟合能力强，对噪声数据鲁棒性好（提供稳定性）
- **梯度提升树**：精度高，能更好地捕捉数据中的复杂模式（提供准确性）
- **集成方式**：软投票（soft voting），基于概率加权，比硬投票更稳定

### 决策2：数据预处理策略
**问题**：原始数据存在哪些质量问题？
- 分类特征（visit_frequency, satisfaction_level, gender）为字符串类型
- 包含年龄<18的无效数据
- 存在缺失值

**设计决策**：
- **分类特征编码**：使用有序编码而非独热编码，保留特征的顺序信息（如visit_frequency的频率高低）
- **过滤规则**：仅保留年龄≥18的有效数据（目标客户群体）
- **缺失值处理**：直接删除缺失值，简化实现并保证数据质量

### 决策3：API设计原则
**问题**：如何设计一个可维护、可扩展的API？

**设计原则**：
- **RESTful风格**：使用标准HTTP方法（GET/POST）和响应状态码（200/400/500）
- **错误处理**：统一错误格式，提供清晰的错误信息（字段缺失、类型错误等）
- **模型信息暴露**：提供模型类型、算法组成等元数据，便于监控和调试
- **扩展性**：接口设计考虑未来扩展，如批量预测、模型更新等

### 决策4：文件拆分策略
**问题**：如何平衡代码复用和职责单一？

**设计决策**：
- **model_training.py**：专注于模型训练和持久化，不包含评估或服务逻辑
- **model_evaluation.py**：专注于模型性能评估，不包含训练或服务逻辑
- **api_service.py**：专注于API服务，不包含训练或评估逻辑

**好处**：
- 每个文件职责单一，易于理解和维护
- 可以独立运行和测试每个模块
- 便于团队协作开发
